# Taste Evolution and Revision Audit

## Purpose

Update the skill from durable taste data: your revisions, co-author comments, reviewer comments, paper analyses, and writing failures encountered during real work.

## When to Use

Use for revision histories, reviewer letters, co-author comments, git diffs, Overleaf changes, paper taste notes, and natural-language requests to improve the skill.

## Inputs Needed

- Comments, diffs, commit logs, paper notes, or examples.
- Before/after text when available.
- The user's judgment about which change was good or bad.

## Procedure

For revision audits:

1. List changes with date/source, files or sections affected, and rough size.
2. Filter out merge commits, PDF-only syncs, duplicate uploads, and logistics.
3. Inspect actual text differences when available.
4. Classify each substantive change as identity, substantive contribution, positioning, register, calibration, or cleanup.
5. Extract durable rules only from identity, substantive contribution, positioning, register, or calibration changes.

For top-paper taste distillation:

1. Store compact notes, not full copyrighted paper text.
2. Extract publication logic, introduction architecture, contribution framing, theorem-to-insight style, experiment narrative, and credibility moves.
3. Merge repeated patterns into the relevant module only after they appear useful across papers.
4. Keep project-specific taste separate from general OM/OR taste.

## Expected Output

Return:

- Durable taste rules.
- Which module should be updated.
- Proposed wording for the update.
- Risks of overfitting to one paper or one co-author comment.

## Common Failure Modes

- Treating grammar cleanup as intellectual progress.
- Copying a paper's prose instead of distilling its structure.
- Overfitting to one famous professor's style.
- Mixing project-specific terminology into the general skill.
